# 语言功能完全实现计划（对齐参考项目 sysml-2ls）

**概述**：对齐参考项目 sysml-2ls，实现最完整语言能力：AST+Scope+引用解析、语义校验、文档符号层级、符号感知重命名、AST 格式化/折叠/语义高亮、补全/悬停/跳转/引用语义化，多文件工作区，以及 LSP 为能力唯一源（Connection 注册全部能力，Monaco 仅 UI）。参考项目：[sensmetry/sysml-2ls](https://github.com/sensmetry/sysml-2ls)；G4 首版来源：[dukoka/sysml-v2-lsp/grammar](https://github.com/dukoka/sysml-v2-lsp/tree/main/grammar)。

---

## 现状与目标

当前部分功能为“简单实现”，且与参考项目 sysml-2ls 的差距包括：无 Scope/引用解析、无语义校验、补全/悬停/跳转/引用非语义化、无多文件、LSP 仅用于诊断。本计划实现**最完整**语言能力，对齐参考项目的架构与能力：**Scope + Linker/References、ValidationRegistry、WorkspaceManager/IndexManager、所有能力经 LSP Connection 暴露、Monaco 仅作 UI**。

---

## G4 语法与校验（首版来源）

- **G4 存放目录**：项目内 **`src/grammar/g4`** 专用于存放 ANTLR 语法文件；仅需将 G4 放入该目录即可作为“G4 校验”的语法来源。
- **首版 G4 来源**：[dukoka/sysml-v2-lsp 的 grammar 目录](https://github.com/dukoka/sysml-v2-lsp/tree/main/grammar)，包含：
  - `SysMLv2Lexer.g4`
  - `SysMLv2Parser.g4`
  - `SysMLv2Lexer.tokens`（由 ANTLR 生成，可选随 Lexer 一并放入或由本地生成）
- **首版做法**：从上述仓库将 `SysMLv2Lexer.g4`、`SysMLv2Parser.g4` 拷贝至本项目的 `src/grammar/g4`，后续通过 ANTLR 生成解析器（如 TypeScript/JavaScript），在校验流程中增加“G4 解析”通道：对同一文档用 Langium 解析（供 AST、补全等）；**开启 G4 时为独立 G4 严格模式**，即用 G4 解析器单独跑一遍，结果单独展示，不合并进主诊断。
- **与现有 Langium 的关系**：当前解析与 IDE 能力仍以 Langium（`src/grammar/*.langium`）为主；G4 首版仅用于“基于 G4 的校验”这一路，规则对照可维护在 `docs/grammar-mapping.md`（G4 规则名 ↔ Langium 规则名）。

**后续如何方便切换**（实现时按此设计，切换仅改配置或目录，不改业务逻辑）：

| 切换目的 | 操作 | 说明 |
|----------|------|------|
| 更换 G4 来源（如换另一仓库的 G4） | 只替换 `src/grammar/g4` 下的 `.g4` 文件，重新执行 ANTLR 生成脚本 | 无需改业务代码；生成产物统一输出到约定目录（如 `src/grammar/g4/generated`） |
| 在「Langium 为主 / G4 为主」之间切换 | 改**单一配置项**（如 `grammarSource: 'langium' | 'g4'` 或环境变量 `GRAMMAR_SOURCE`） | 解析/校验入口统一从该配置读取；切换时只改配置不改代码 |
| 仅开关 G4 校验（解析仍用 Langium） | 改配置项（如 `g4Validation: true | false`） | 关闭则诊断只保留 Langium 结果；**开启则为独立 G4 严格模式**（G4 解析结果单独展示，不合并进主诊断） |

建议：配置集中放在一处（如 `langium-config.json` 同级或 `src/grammar/config.ts`），文档中注明“解析/校验来源与 G4 开关”的配置项名与可选值，便于后续维护和切换。

## 1. AST 与范围基础设施

所有“完全实现”依赖两件事：**AST 结构**（已有，`src/grammar`）和**节点在文档中的范围**（行/列）。

- **现状**：`parseSysML` 返回根 AST；未暴露 Langium Document，AST 节点上是否有 `$cstNode`/range 需在运行时确认。
- **方案**：若 Langium 解析结果中节点带 range，在 `src/grammar` 下新增 `astUtils.ts`，提供 `getNodeRange(node)`、`getDocumentSymbols(root)` 等；若不提供则采用“结构用 AST、位置用现有数据”的混合方案，在 `astUtils.ts` 中统一封装。
- **交付**：`src/grammar/astUtils.ts`（或等价模块），提供“从 AST 取结构 + 取 range”的单一入口。

---

## 2. 语义层：Scope 与引用解析（对齐参考 references）

- **目标**：在解析后的 AST 上建立 Scope；对类型引用、特征引用等做引用解析（resolve）。
- **实现**：ScopeProvider、引用解析，输出供跳转定义、查找引用、重命名、补全使用。
- **涉及**：新建 `scope.ts` / `references.ts`，与 astUtils 和解析器配合。

---

## 3. 语义校验（对齐参考 DocumentValidator + ValidationRegistry）

- **目标**：解析成功后运行语义规则，将结果并入诊断。
- **实现**：ValidationRegistry 注册多条校验规则；解析成功后遍历 AST/Scope，收集违规并转为 LSP Diagnostic。
- **涉及**：新建 `validation/` 或 `semanticValidation.ts`，在 Worker 的 validateDocument 及 validator 回退路径中追加语义诊断。

---

## 4. 文档符号（层级大纲）

- **目标**：大纲为层级结构（package → part def → 成员），与 AST 一致。
- **实现**：astUtils 中 `astToDocumentSymbols(root)`；index 的 provideDocumentSymbols 解析成功时调用，失败时回退现有逻辑。
- **涉及**：documentSymbols.ts、index.ts、astUtils.ts。

---

## 5. 跳转定义与查找引用（基于引用解析，支持跨文件）

- **目标**：Go to Definition 跳转到解析到的定义；Find References 列出所有解析到当前定义的引用。
- **实现**：依赖引用解析；单文件阶段可先只做同文件，多文件阶段接入 Index 与跨 URI。
- **涉及**：scope.ts/references.ts、DefinitionProvider/ReferenceProvider；若 LSP 为唯一源则由 Worker 实现。

---

## 6. 重命名（符号感知，对齐参考 ExecuteCommand）

- **目标**：仅重命名当前符号的定义及其解析到的所有引用，不误改同名异义。
- **实现**：使用引用解析得到“该定义 + 其引用”，生成 WorkspaceEdit。
- **涉及**：references.ts、RenameProvider 或 LSP textDocument/rename。

---

## 7. 补全与悬停的语义化（对齐参考 CompletionProvider + HoverProvider）

- **补全**：根据光标位置得到当前 Scope；在类型位置只补类型，在成员位置补成员、关键字等。
- **悬停**：根据光标定位到 AST 节点，显示元素类型、声明名、可选文档。
- **实现**：依赖 astUtils + Scope；completion.ts 用 scope 可见符号 + 上下文过滤；HoverProvider 解析并定位节点后生成 markdown。

---

## 8. 格式化（AST 感知）

- **目标**：按 AST 元素边界做缩进与换行（package/part/port 块内统一缩进、可选属性对齐）。
- **实现**：解析成功时用 AST 得到每行缩进等级，解析失败时保持现有逻辑。
- **涉及**：formatter.ts、astUtils。

---

## 9. 语义高亮（AST 驱动）

- **目标**：基于 AST 区分 namespace、type、class、property、keyword 等。
- **实现**：解析成功时遍历 AST 用 getNodeRange 映射 token 类型；解析失败时回退现有正则。
- **涉及**：semanticTokens.ts、astUtils。

---

## 10. 折叠（AST 驱动）

- **目标**：折叠区域与 AST 块一致（package/part def/port def 等体的 `{` 到 `}`）。
- **实现**：解析成功时从 AST 收集有 body 的命名空间节点生成 FoldingRange；解析失败时保留现有括号+关键词逻辑。
- **涉及**：index.ts 的 registerFoldingRangeProvider、astUtils。

---

## 11. 签名帮助（扩展）

- **目标**：覆盖更多标准库/内置调用；可选从 AST 识别调用式并给出占位签名。
- **实现**：BUILTIN_SIGNATURES 扩展一批常见内置（toInteger、toReal、size、empty 等）；可选 AST 解析 InvocationExpression 改进 activeParameter。
- **涉及**：index.ts。

---

## 12. 代码操作（扩展）

- 在现有 4 类快速修复基础上，增加未闭合字符串/括号、重复定义提示（带跳转）等；可选 AST 重构类 CodeAction。
- **涉及**：index.ts 的 provideCodeActions。

---

## 13. 多文件与工作区（对齐参考 WorkspaceManager + IndexManager）

- **目标**：支持多文档（虚拟 URI），Worker 内维护文档索引；引用解析与跳转/引用/重命名可跨文件。
- **实现**：虚拟工作区（Map<uri, text>）+ 每文档 Index；Scope/引用解析查询 Index 实现跨文件 resolve。
- **涉及**：Worker 文档存储与索引、scope.ts/references.ts 跨 URI、可选多 tab/多 URI UI。

---

## 14. LSP 为能力唯一源（对齐参考 Connection + SysMLLanguageServer）

- **目标**：补全、悬停、跳转、引用、重命名、格式化、语义高亮、诊断、代码操作等全部经 LSP 提供；Monaco 仅作 UI。
- **实现**：Worker 内 createConnection 并注册所有 LSP 请求的 handler；编辑器侧 lspClient 与 Worker 通过 LSP 协议通信，Monaco 仅用 LSP 结果展示。
- **涉及**：sysmlLSPWorker.ts、lspClient.ts、CodeEditor.tsx、index.ts。

## 15. 实施顺序建议

| 阶段 | 内容 | 依赖 |
|------|------|------|
| 1 | AST 范围基础设施（astUtils） | 无 |
| 2 | 语义层：Scope + 引用解析 | astUtils |
| 3 | 语义校验（ValidationRegistry） | AST + Scope |
| 4 | 文档符号层级、折叠/语义高亮/格式化 AST 化 | astUtils |
| 5 | 跳转定义/查找引用/重命名 基于引用解析 | Scope/引用解析 |
| 6 | 补全/悬停 语义化（Scope + AST） | Scope、astUtils |
| 7 | 签名帮助与代码操作扩展 | 无 |
| 8 | 多文件与工作区（Index、跨文件） | Scope、引用解析 |
| 9 | LSP 为能力唯一源（Connection 全注册、Monaco 仅 UI） | 以上全部 |
| （可与阶段 3/4 或任意阶段并行） | G4 接入：拷贝 G4 至 `src/grammar/g4`、ANTLR 生成解析器、独立 G4 严格模式展示 | 无（与 Langium 解析/Scope 无依赖） |

**流程与依赖确认**（已核对，可依此执行）：
- **阶段 1 为根**：astUtils 不依赖其他计划内模块，必须先做。
- **阶段 2 依赖 1**：Scope/引用解析依赖 astUtils 与解析器（§2）；阶段 3、5、6、8 均依赖阶段 2。
- **阶段 3 依赖 1+2**：语义校验在解析成功后跑，需 AST + Scope（§3）。
- **阶段 4 依赖 1**：文档符号/折叠/语义高亮/格式化均只需 astUtils + 解析器（§4、8、9、10），不依赖 Scope。
- **阶段 5、6 依赖 2（及 1）**：跳转/引用/重命名依赖引用解析（§5、6）；补全/悬停依赖 Scope + astUtils（§7）。
- **阶段 7 无前置**：签名帮助与代码操作（§11、12）可不依赖 Scope，与 1～6 无强顺序。
- **阶段 8 依赖 2**：多文件/工作区在 Scope 与引用解析基础上扩展 Index 与跨 URI（§13）。
- **阶段 9 依赖 1～8**：LSP 为唯一源时，所有能力迁入 Worker，故需前述能力均已实现（§14）。
- **G4 行**：独立于 Langium 解析与 Scope，可与任意阶段并行；仅需 ANTLR 与 G4 文件，结果单独展示。

## 16. 验收要点（逐项清点）

实施验收时按下列项逐项测试并勾选。

### 16.1 诊断
- [ ] 解析优先：解析成功且无 parser/lexer 错误时，不再跑正则校验，无误报。
- [ ] VehicleExample 等示例中：注释、`part name : Type`、`port name : Type`、`in/out attribute`、`Wheel[4]` 等不被误报。
- [ ] 语义错误（未定义引用、类型不匹配等）由 ValidationRegistry 产出并正确展示。

### 16.2 文档符号
- [ ] 大纲为 package → part/port def → 成员层级，与 AST 一致。
- [ ] 解析失败时回退到现有扁平/符号列表行为。

### 16.3 跳转定义 / 查找引用 / 重命名
- [ ] 跳转定义：跳转到解析到的定义（同文件；可选跨文件）。
- [ ] 查找引用：列出所有解析到当前定义的引用。
- [ ] 重命名：仅修改该符号的定义与引用，不误改同名异义。

### 16.4 补全与悬停
- [ ] 补全：基于 Scope + AST，类型与成员列表正确，上下文过滤合理。
- [ ] 悬停：显示元素类型、声明名、可选文档等信息。

### 16.5 格式化
- [ ] 块缩进与 AST 一致；可选属性对齐。

### 16.6 语义高亮与折叠
- [ ] 语义高亮：与 AST 一致，解析失败时回退到现有行为。
- [ ] 折叠：折叠区与 AST 块边界一致。

### 16.7 签名帮助与代码操作
- [ ] 签名帮助：至少增加一批内置函数签名；可选更准的 activeParameter。
- [ ] 代码操作：至少再增加 2～3 类可自动修复的 diagnostic 对应操作。

### 16.8 LSP 为能力唯一源（若实施阶段 9）
- [ ] 所有能力（补全、悬停、跳转定义、查找引用、重命名、格式化、语义高亮、诊断、代码操作等）经 LSP 提供。
- [ ] Monaco 仅作 UI，不保留“本地 Provider 为主”的双轨。

### 16.9 G4 语法与独立严格模式
- [ ] `src/grammar/g4` 已存在，且含首版 G4（如 SysMLv2Lexer.g4、SysMLv2Parser.g4）。
- [ ] 开启 G4 时为**独立 G4 严格模式**：G4 解析结果单独展示（如独立诊断源/面板或标签），不合并进主 Langium 诊断。
- [ ] 解析/校验来源与 G4 开关的配置项已集中并文档化（配置项名与可选值可在文档中查到）。
