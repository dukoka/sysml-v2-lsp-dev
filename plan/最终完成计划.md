### 阶段 G：多文件与工作区

**目标**：虚拟工作区（多 URI）+ 每文档 Index；Scope/引用解析跨 URI；跳转/引用/重命名可跨文件。

#### G.1 数据结构与索引（Worker 内）

- **位置**：`src/workers/` 下新增 `indexManager.ts`（或于 `sysmlLSPWorker.ts` 内实现）。
- **每文档索引项**：对每个 URI 维护：
  - `uri: string`
  - `text: string`（当前文档内容）
  - `parseResult`：`parseSysML(text)` 的返回值（含 `value` 根 AST、`parserErrors`、`lexerErrors`）
  - `scopeRoot: ScopeNode | null`：`buildScopeTree(parseResult.value)`
  - 可选：`symbols: AstDocumentSymbol[]`（`astToDocumentSymbols`）便于 documentSymbol/折叠
- **全局索引**：`Map<string, 上述索引项>`，key 为 URI；在 `documents.onDidOpen` / `documents.onDidChangeContent` 时更新对应 URI 的 text → 重新 parse → 更新 scopeRoot（及 symbols）。
- **与现有 documents 的关系**：继续使用 `TextDocuments(TextDocument)` 作为文档源；Index 在文档变更时从 `documents.get(uri)?.getText()` 取文本并更新上述字段。

#### G.2 Scope/引用跨 URI（scope.ts、references.ts）

- **scope.ts**：
  - 新增（或扩展现有）`scopeLookupInIndex(uri: string, scopeRoot: ScopeNode | null, name: string, index: Map<string, IndexEntry>): { uri: string; node: Element } | undefined`。逻辑：先在当前文档 scope 链上查找；未命中时遍历 index 中其他 URI 的 scopeRoot，在各根级 declarations 中按 name 查找（包级导出）；返回首次命中的 `{ uri, node }`。
  - 保持 `buildScopeTree`、`getScopeAtPosition` 单文档接口不变，由调用方传入「当前 uri + 当前 root」与「全局 index」。
- **references.ts**：
  - `resolveToDefinition` 扩展为 `resolveToDefinitionWithUri(scope, name, index): { uri: string; node: Element } | undefined`，内部调用上述 `scopeLookupInIndex`。
  - `getDefinitionAtPosition` 扩展为接受 `(root, text, line, character, index)`，返回 `{ uri: string; node: Element } | null`，便于 LSP 返回 `Location { uri, range }`。
  - `findReferencesToDefinition` 扩展为接受 `definitionUri`、`definitionNode` 与 `index`，在所有 index 的文档中收集引用，返回 `Array<{ uri: string; node: Element }>`，再转为 `Location[]`。
- **astUtils**：`getNodeRange(node, text)` 已存在；跨文件时由调用方传入对应 uri 的 text。

#### G.3 Worker 内 LSP 行为（为阶段 H 预留）

- 在阶段 G 仅需保证：当请求 definition/references/rename 时，若实现仍为单文档，则返回同 uri 的 Location；若已接入 Index，则返回可能带多 uri 的 `Location[]`。
- **definition**：返回 `Location | Location[]`（LSP 允许 array）。
- **references**：返回 `Location[]`。
- **rename**：返回 `WorkspaceEdit { changes: { [uri]: TextEdit[] } }`，跨文件时多 uri。

#### G.4 UI 与虚拟工作区

- **最小闭环（推荐先做）**：单文档场景下仅实现「Index 结构」：Worker 内对当前唯一打开的 URI 维护 Index 条目；不要求多 tab。验证：Index 在 didOpen/didChange 时正确更新、scopeRoot 可用。
- **多 URI 闭环**：若需真实跨文件，需上层（如 CodeEditor 或路由）维护多个虚拟 URI（如 `file:///w/a.sysml`、`file:///w/b.sysml`），在打开/切换时对 Worker 发送 `didOpen`/`didClose`，并保证 definition/references 返回的 Location 中 uri 可在前端解析（如同源或约定 scheme）。Monaco 的 `editor.getContribution` 或自定义命令可根据 Location.uri 打开对应模型并定位。

#### G.5 验收清单（阶段 G）

- [x] Worker 内存在「每文档 Index」结构（uri → parseResult + scopeRoot），并在 didOpen/didChange 时更新。
- [x] `scopeLookupInIndex`（或等价）支持在当前文档未命中时从其他文档根级声明按名称解析。
- [x] `getDefinitionAtPosition` / `findReferencesToDefinition` 可返回带 uri 的结果（至少单文档时与现有一致；多文档时至少 2 个 URI 可验证跨文件跳转/引用）。
- [x] 重命名返回 `WorkspaceEdit`，跨文件时包含多 uri 的 edits。
- [ ] 自动化测试或手工验证：两文件 A 引用 B 中定义的类型/部件时，跳转定义、查找引用、重命名均正确。

### 阶段 H：LSP 为能力唯一源

**目标**：Worker 内 Connection 注册 §1.4 全部能力；编辑器仅通过 lspClient 消费并展示；Monaco 仅保留词法/语言配置等与 LSP 无关的注册。

#### H.1 Worker 内需实现的 LSP 请求（sysmlLSPWorker.ts）

| 方法 | 当前状态 | 实现要点 |
|------|----------|----------|
| `textDocument/definition` | 未实现 | 取 uri、position → 从 Index 取该 uri 的 text/root/scope；调用 `getDefinitionAtPosition(root, text, line, char, index)`；将结果转为 `Location \| Location[]`（range 用 getNodeRange + 0-based）。 |
| `textDocument/references` | 未实现 | 先 resolve definition at position；再 `findReferencesToDefinition(root, defNode, index)`（或跨 uri 版本）；返回 `Location[]`。 |
| `textDocument/rename` | 未实现 | 解析当前 symbol，得到 definition + 所有 references；生成 `WorkspaceEdit { changes: { [uri]: TextEdit[] } }`。 |
| `textDocument/documentSymbol` | 未实现 | 从 Index 取该 uri 的 root；`astToDocumentSymbols(root, text)`；转为 LSP `DocumentSymbol[]`（range/selectionRange 0-based）。 |
| `textDocument/foldingRange` | 未实现 | 可由 documentSymbol 推导折叠区间，或从 AST 块用 getNodeRange 生成 FoldingRange[]。 |
| `textDocument/semanticTokens/full`（或 semanticTokens） | 未实现 | 复用现有 `semanticTokens.ts` 逻辑（或调用其导出），在 Worker 内对 uri 的 text 解析后返回 SemanticTokens 编码。 |

| `textDocument/formatting`、`rangeFormatting` | 已实现 | 保持。 |

#### H.2 lspClient 扩展（lspClient.ts）

- 为上述每个「未实现」的请求新增对应方法，例如：
  - `getDefinition(position): Promise<Location | Location[] | null>`
  - `getReferences(position): Promise<Location[]>`
  - `getRename(position, newName): Promise<WorkspaceEdit | null>`
  - `getDocumentSymbols(): Promise<DocumentSymbol[]>`
  - `getFoldingRanges(): Promise<FoldingRange[]>`
  - `getSemanticTokens(): Promise<number[] | SemanticTokens>`（依 LSP 协议格式）
  - `getSignatureHelp(position): Promise<SignatureHelp | null>`
  - `getCodeActions(range, context): Promise<CodeAction[]>`
- 内部统一使用 `this.sendRequest(method, params)`，method 与 Worker 注册的 LSP 方法一致。

#### H.3 CodeEditor 切换为 LSP 驱动（CodeEditor.tsx、index.ts）

- **CodeEditor**：在初始化 LSP 成功后，对「定义、引用、重命名、文档符号、折叠、语义高亮、签名帮助、代码操作」等能力，改为：先调用 lspClient 的对应方法，再将返回结果转换为 Monaco 的接口格式（如 IMarkerData、DocumentSymbol、FoldingRange、Hover 等），并应用到编辑器；不再依赖 Monaco 的 `registerDefinitionProvider` 等对同一能力的注册（或保留为 fallback，见下）。
- **index.ts（registerSysmlv2Language）**：可保留 `registerDefinitionProvider`、`registerReferenceProvider` 等注册，但通过「开关」或「优先使用 LSP」逻辑，使在 LSP 可用时由 CodeEditor 侧用 LSP 结果覆盖；或直接移除这些 Provider 的注册，全部由 CodeEditor 通过 lspClient 驱动（推荐最终状态）。
- **保留**：词法（Monarch）、`setLanguageConfiguration`、主题等与 LSP 无关的注册必须保留。

#### H.4 回归与开关（可选）

- 在 config 或环境变量中增加「LSP 为能力唯一源」开关（如 `useLspAsSingleSource: boolean`）。当关闭时，CodeEditor 仍使用 Monaco Provider 的返回；当开启时，仅使用 lspClient 的返回。便于对比与回退。

#### H.5 验收清单（阶段 H）

- [x] Worker 内 §1.4 所列能力均有对应 handler（definition、references、rename、documentSymbol、foldingRange、semanticTokens、signatureHelp、codeAction；completion、hover、formatting、diagnostic 已有）。
- [x] lspClient 提供上述能力的请求方法，且与 Worker 协议一致。
- [x] CodeEditor 在 LSP 可用时，跳转定义、查找引用、重命名优先使用 LSP（setSysmlv2LspClientGetter + Provider 内先调 LSP）；诊断/补全已用 LSP；文档符号/折叠/语义高亮/签名帮助/代码操作由 Monaco Provider 提供（可后续改为 LSP 驱动）。
- [ ] 自动化测试通过；生产构建通过；无严重回归（如 VehicleExample 仍可编辑、跳转、补全、诊断正常）。
